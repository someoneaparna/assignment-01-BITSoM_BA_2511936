## Database Recommendation

For a patient management system, I would recommend MySQL as the primary system of record. Healthcare workflows depend on strong consistency, transactional correctness, and auditable updates. Patient demographics, appointments, prescriptions, billing entries, and clinical events are tightly related and often updated together. In that context, ACID guarantees matter more than schema flexibility. A relational model also makes it easier to enforce constraints such as valid patient identifiers, referential integrity between visits and treatments, and well-structured access controls.

CAP theorem trade-offs also support that choice. A hospital should favor consistency over loose eventual consistency for core clinical operations. If two clinicians or back-office systems update the same patient workflow, the organization needs one trusted truth immediately, not a BASE-style model where changes reconcile later. MongoDB can scale well and handle flexible documents, but the main patient ledger should not optimize first for schema drift when correctness is the bigger business risk.

My answer changes only partially if the startup also needs a fraud detection module. I would still keep MySQL as the transactional backbone, but I would add a separate analytical or event-driven component for fraud signals. Fraud detection often benefits from high-volume event ingestion, semi-structured feature storage, and near-real-time scoring. That module could use MongoDB, Kafka, or a feature store alongside the relational OLTP database. In other words, I would not replace MySQL with MongoDB for the whole platform; I would use polyglot persistence. MySQL remains the authoritative source for patient workflows, while a more flexible secondary system can support behavioral analytics, anomaly detection, and model experimentation.
